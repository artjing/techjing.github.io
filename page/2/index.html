重构：利用objction实现组件化开发

post @ 2016-05-20

重构：利用objction实现组件化开发

先从为什么要使用组件化开发说起

下面是一个很常规的需求，我们要在首页添加一个Tabbar控制器，里面有几个模块可以来回切换。为了实现这个功能，我们必须在delegate中导入所有需要的Controller的头文件（假如还有第3、4、5个控制器，都要加进来）。先看下面的实现代码： 

#import "AppDelegate.h"
#import "FirstViewController.h"
#import "SecondViewController.h"

@interface AppDelegate ()
@end
@implementation AppDelegate

- (BOOL)application:(UIApplication *)application didFinishLaunchingWithOptions:(NSDictionary *)launchOptions {

self.window = [[UIWindow alloc] initWithFrame:[[UIScreen mainScreen] bounds]];
self.window.backgroundColor = [UIColor whiteColor];
[self.window makeKeyAndVisible];

// 创建tabbar控制器
UITabBarController * tabbarVC = [[UITabBarController alloc] init];
self.window.rootViewController = tabbarVC;

// 创建第一个控制器
FirstViewController * firstViewController = [[FirstViewController alloc] init];
UINavigationController * firstNav = [[UINavigationController alloc] initWithRootViewController:firstViewController];
firstViewController.tabBarItem = [[UITabBarItem alloc] initWithTitle:@"button1" image:[UIImage imageNamed:@""] tag:1001];
firstViewController.view.backgroundColor = [UIColor grayColor];
firstViewController.title = @"第一个控制器";

// 创建第二个控制器
SecondViewController * secondViewController = [[SecondViewController alloc] init];
UINavigationController * secondNav = [[UINavigationController alloc] initWithRootViewController:secondViewController];
secondViewController.tabBarItem = [[UITabBarItem alloc] initWithTitle:@"button2" image:[UIImage imageNamed:@""] tag:1002];
secondViewController.view.backgroundColor = [UIColor greenColor];
secondViewController.title = @"第二个控制器";

tabbarVC.viewControllers = @[firstNav,secondNav];
[self.window makeKeyAndVisible];

return YES;
}

这段代码是我们常规开发的做法，简单明了，在需要的地方创建和调用我们需要的对象，对于复杂度不高的项目，这样做很好，既有较高的可读性，维护成本也低。但对于大的项目来说，APP中都有很多个功能模块，模块之间可以相互调用和通信，当我们的项目复杂度越来越高，模块之间的依赖也越强，很难分割。为了解决这个问题，很多人开始将项目重构，做组件化的开发，模块与模块之间解耦，从面向对象到面向接口的编程。这样4A模块可以交给A组去做，B模块交给B组去做，其它的模块也可以进行任务划分。

想要实现这样的目的，我们要如何才能做到解除各组件的依赖，又能调用到组件暴露出来的接口方法呢？我们需要一个中间层，负责转发信息，可以叫它Mediator、Manager或者Router都可以。下面我们将使用objection库对我们刚才的代码进行重构。

使用objection库对项目进行重构

1，定义我们需要的协议类

将我们用到的所有protocal及接口都放到protocals.h中。
项目的目录如下：

AppDelegate中，只依赖协议文件protocals.h和Objection.h，后面我们将利用protocal获得我们需要的对象，以及调用对象的接口。对象的接口也定义在protocals.h文件中。

#import <Foundation/Foundation.h>
@protocol protocals <NSObject>
@end

// FirstViewController的protocal
@protocol firstViewControllerProtocals <NSObject>
-(void)configWithTag;
@end

// SecondViewController的protocal
@protocol secondViewControllerProtocals <NSObject>
-(void)configWithTag;

@end
2，定义module

定义module，继承自JSObjectionModule，在里面实现两个方法+(void)load和-(void)configure。

#import "firstModule.h"
#import "FirstViewController.h"
#import "JSObjection.h"

@implementation firstModule

+(void)load
{
JSObjectionInjector * injector = [JSObjection defaultInjector];
injector = injector?:[JSObjection createInjector];
injector = [injector withModule:[[self alloc] init]];
[JSObjection setDefaultInjector:injector];
}

-(void)configure
{
[self bindClass:[FirstViewController class] toProtocol:@protocol(firstViewControllerProtocals)];
}
3，调用接口

接下来就是对文章最开始那段代码的重构，通过objection，我们可以用prorocal直接获得对象，而不需要再创建。

#import "AppDelegate.h"
#import "protocals.h"
#import "Objection.h"

@interface AppDelegate ()

@end

@implementation AppDelegate

- (BOOL)application:(UIApplication *)application didFinishLaunchingWithOptions:(NSDictionary *)launchOptions {

self.window = [[UIWindow alloc] initWithFrame:[[UIScreen mainScreen] bounds]];
self.window.backgroundColor = [UIColor whiteColor];
[self.window makeKeyAndVisible];
[self setupTab];
return YES;

}

-(void)setupTab
{

// 创建tabbar控制器
UITabBarController * tabbarVC = [[UITabBarController alloc] init];
self.window.rootViewController = tabbarVC;

// 创建第一个控制器

// 调用接口，可以将上面初始化的方法写在里面。

// 创建第二个控制器
UIViewController<secondViewControllerProtocals> * secondViewController = [[JSObjection defaultInjector] getObject:@protocol(secondViewControllerProtocals)];
// 调用接口
[firstViewController configWithTag];
UINavigationController * secondNav = [[UINavigationController alloc] initWithRootViewController:secondViewController];

tabbarVC.viewControllers = @[firstNav,secondNav];
[self.window makeKeyAndVisible];
}

HTTPS及RSA加密

post @ 2016-05-20

最近在苏宁的项目中，有一个需要在客户端加签的需求，后台只提供了我一串公钥的字符串，刚开始拿到这个完全是一头雾水，于是乎决定彻底的将HTTPS以及RSA加密进行了一遍梳理，顺便记录下来。分享给有需求的小伙伴们。

HTTPS与SSL

HTTP与HTTPS的区别

HTTPS本质上也是HTTP，只不过它添加了更安全的SSL（Secure Socket Layer），一个用于保障通信安全的层。

总结一下HTTP与HTTPS的区别：

1，HTTPS需要使用正规机构颁发的CA证书。
2，HTTP是超文本传输协议，信息明文传输。HTTPS是安全的SSL加传输协议。
3，HTTP与HTTPS的连接方式、端口不一样，HTTP的端口是80，HTTPS的端口是443。

SSL/TLS的运行机制

不使用SSL的HTTP通信，也就是不加密的通信，它存在三个风险：

1，窃听风险，第三方可以获取通信的内容。
2，篡改风险，通信内容可以被修改。
3，冒充风险，冒充身份进行通信。
SSL正是为了解决这三大风险而设计，它拥有：加密传输、校验机制和身份认证。

SSL协议的握手过程

SSL加密通信，在开始加密通信之前，客户端和服务器首先必须建立连接和交换参数，也就是握手的过程。握手的过程分为五步：
1，Client向Server发送请求，并发送协议版本号、客户端生成的随机数、支持的加密方式。
2，Server确认双方使用的加密方式，发送Server生成的随机数和数字证书。
3，Client确认数字证书的有效性，再生成一个新的随机数，并使用有效的证书加密这个随机数，发送给Server。
4，Server使用自己的私钥，解密Client发来的随机数。
5，Client和Server根据约定的加密方法，使用前面的三个随机数，生成对话秘钥，用来加密接下来的整个对话过程，对话秘钥是对称加密。而服务器的公钥放在Server的数字证书中，它的公钥和私钥只用于加密和解密这个对话秘钥。

（对称加密：秘钥只有一个，加密解密使用同一个，典型的加密算法有DES和AES。
非对称加密：有一个公钥和私钥，加解密使用不同的秘钥，公钥加密要用私钥解密，私钥加签，需要用公钥解签。典型的非对称加密有RSA、DSA。）

在IOS开发过程中，如果请求接口为HTTPS，而服务器返回来的证书是自签名证书，网络连接会发生错误。出于安全性考虑，我们最好使用正规机构颁发的CA证书，接收任何机构颁发的证书将会危害数据的完整性和安全性。在没有合法证书的情况下，为了使用HTTPS，我们可以手动的关掉证书验证： 

#if KAllowInvalidHttps
request.validatesSecureCertificate = YES;
#else
request.validatesSecureCertificate = YES;
#endif 
// 在配置文件中，添加一个宏定义，可以定义打开或者关系，在发送网络请求时，选择是否忽略安全证书。

如何添加双向认证

上面的SSL握手里提到客户端验证服务器证书是否合法的过程，对于双向认证，顾名思义，也就是服务器也要确认客户端是否合法。Server要验证Client发送的证书，如果没有验证通过，则拒绝连接，这样做无疑是更安全的。它的过程如下：
首先服务器提供一个ssl证书（提供给客户端用的是.crt文件），最后要转成.cer

RSA加密算法

RSA公钥加密体制包含如下三个算法：
1，秘钥生成算法（ketgen）
2，加密算法（encrypt）
3，解密算法（decrypt）
关于算法的具体描述，这里不详细解释，推荐一篇极好的博客：

加签

RSA的签名算法：
1，秘钥生成算法（ketgen）
2，加签算法（sign）
3，验签算法（verify）

一些相关的名词解释

OpenSSL

它是SSL的一个实现。

X509证书

X509是证书的一种标准，它定义了证书中应该包含哪些内容，共包含两种编码格式：

PEM: ———begin—————end———— 作为开头和结尾，采用Base64编码。

DER: 二进制编码。

CER、CRT：还有其它的证书格式例如：CER、CRT(certification)。可能是PEM或者DER，可打开查看格式知道。

CSR: 非证书，证书签名请求，例如想苹果请求开发者证书时使用。

P12证书：

是PKCS12的简称，PKCS是一个证书标准的缩写。P12将CRT和KEY存放在一起，用于存放个人证书或者私钥，有一个提取秘钥，通常使用DER编码，所以我们在自己生成P12证书的时候，先将PEM证书转换成DER证书，再生成P12证书。

生成P12的方法：

先用openssl创建CA证书，也就是PEM文件。再用openssl将PEM文件转成DER文件。最后用X509转为PKCS12格式的证书，其包含公钥/私钥及其证书。

走进RunLoop

post @ 2016-05-20

当我们用Xcode新建一个工程，并运行这个工程，你有没有思考过，程序为什么会一直运行着不退出？如果你在页面上再加一个scrollView，当我们滑动页面时，是谁在处理我们的触摸事件呢？今天，我们就要走进这个“我们无时无刻在使用却一直被我们忽视的Runloop”。

RunLoop的定义：

事件处理的循环，用来不停的调度工作以及处理输入事件。

使用RunLoop的目的：

让线程有工作时忙于工作，没工作时处于休眠。如果没有RunLoop，我们的应用程序一启动就会退出，无法进行后面的操作。它就相当于一个循环，处理应用中的各种触摸事件、定时器及selector事件，保证程序的程序运行。

RunLoop的创建

在主线程中，默认的Runloop已经创建，也就是那个能保持程序不退出的RunLoop。一个RunLoop对应一条唯一的线程，如果我们想让子线程不死，就给这条线程开启一个RunLoop。

NSRunLoop
 // 获得主线程的RunLoop
NSRunLoop * mainLoop = [NSRunLoop mainRunLoop];

// CF获得主线程的CFRunLoopRef
CFRunLoopRef * mainLoopRef = CFRunLoopGetMain();

// 开启一个RunLoop。给线程开启RunLoop，不使用init方法，而是使用currentRunLoop,如果不手动开始线程的RunLoop，它默认是不开启的。
NSRunLoop * loop = [NSRunLoop currentRunLoop];    
// CF获得当前的CFRunLoopRef
CFRunLoopRef * loopRef = CFRunLoopGetCurrent();
CFRunLoopTimerRef
// Time事件,它是时间的触发器。
// RunLoop一旦启动，就会选则一种模式，选中之后，就不管其它的模式了，一个mode钟可以添加多个NSTimer。
CFRunLoopTimerRef;

// 调用了scheduledTimerWithTimeInterval，就会自动添加到当前的runloop里，而且runloop的运行模式为KCFRunloopDefaultMode
NSTimer * time = [NSTimer scheduledTimerWithTimeInterval:2.0 target:self selector:@selector(clickCount:) userInfo:nil repeats:YES];
// 更改模式
[[NSRunLoop currentRunLoop] addTimer:time forMode:NSRunLoopCommonModes];


// 定时器加到了NSDefaultRunLoopMode模式下，一旦runloop模式切换，定时器就不工作了
NSTimer * timer = [NSTimer timerWithTimeInterval:2.0 target:self selector:@selector(clickCount:) userInfo:nil repeats:YES];
[[NSRunLoop currentRunLoop] addTimer:timer forMode:NSDefaultRunLoopMode];
Runloop在AFNNetworking中的使用

AFURLConnectionOperation 这个类是基于 NSURLConnection 构建的，其希望能在后台线程接收 Delegate 回调。为此 AFNetworking 单独创建了一个线程，并在这个线程中启动了一个 RunLoop：

+(void)networkRequestThreadEntryPoint:(id)__unused object {
@autoreleasepool {
[[NSThread currentThread] setName:@"AFNetworking"];
NSRunLoop *runLoop = [NSRunLoop currentRunLoop];
[runLoop addPort:[NSMachPort port] forMode:NSDefaultRunLoopMode];
[runLoop run];
}
}

+(NSThread *)networkRequestThread {
static NSThread *_networkRequestThread = nil;
static dispatch_once_t oncePredicate;
dispatch_once(&oncePredicate, ^{
_networkRequestThread = [[NSThread alloc] initWithTarget:self selector:@selector(networkRequestThreadEntryPoint:) object:nil];
[_networkRequestThread start];
});
return _networkRequestThread;
}

RunLoop 启动前内部必须要有至少一个 Timer/Observer/Source，所以 AFNetworking 在 [runLoop run] 之前先创建了一个新的 NSMachPort 添加进去了。通常情况下，调用者需要持有这个 NSMachPort (mach_port) 并在外部线程通过这个 port 发送消息到 loop 内；但此处添加 port 只是为了让 RunLoop 不至于退出，并没有用于实际的发送消息。

读黑川雅之《设计修辞法》

post @ 2016-05-16

最近读了黑川雅之的《设计修辞法》，发现自己对从前无法理解的“减法”的设计，豁然开朗。书中描述了很多日本人的审美意识，让我想起前之前读过的田中一光的《设计的觉醒》，也有很多有意思的、更细致的关于日本审美意识的描述，这可以放到下一篇来介绍。说到今天的主题——《设计修辞法》，这篇文章主要介绍书中作者的一些观点，后面还会接着进行补充。

黑川雅之

黑川雅之出身建筑世家，父亲是著名的世界级建筑大师黑川纪章。黑川雅之从90年代后期多活跃于工业设计，在他的设计中，更多可以看到他对产品空间时间的思考，这可能是源于他建筑的背景。从他的作品中可以看出他对日本传统文化的理解和传承。

藏在意识深处的审美意识

人类自从出现在地球上，便储存着各种各样的记忆。通过将这些记忆放入DNA之后，每个人从降临人世之时便拥有一定的生存智慧，记忆中的一项就是人类共有的语言。有共同的语言，我们就可以明白对方要表达的意思。就如同我们看到火焰后所表现出的欢喜或是恐惧一样，我们将外界事物所具有的意义也一同放入了DNA之中，而这就是隐形意识。隐形意识一般潜藏在人的意识深处，并且会在某一个非特定时间出现。
许多审美意识都被潜藏在了意识深处，它会作为一种共通的感受和理解引起人的共鸣，事实上，每个人都具有独具特色的审美意识，世界上的各个民族也具有独特的审美意识，甚至有一种审美意识是全人类通用的。而在这本《设计修辞法》中，黑川雅之想要探讨的正是用设计表达的意义，引起对方的共鸣，并游走于人类DNA的隐形记忆的普遍性之中。

素材的感觉

生命体无时无刻感受着空气触及皮肤的感觉，用手和脚的神经感受着大地的触觉，用皮肤感受着与同伴的接触，感受着季节和时间的变化，也就是说，生命体的身体感觉向我们传输关于空间和时间的全部信息。
在现代，人们往往将设计归结为形状和色彩的问题。从人类所具有的感知器官来说，视觉确实具备强烈感知能力，当我们思考如何去感受一件物品的形状时，第一个想到的是用眼睛观察，但实际上，利用身体去触摸、感知物品也是非常重要的一种形式。我们可以通过网络传送图像，却无法传递自身的身体感知，比起用眼睛看，存在本身以及用全身去感知它的存也是很重要的，当事情将存在本身所具有的感觉全部传达给人类，如果人类内心对此产生了情感，那事情的存在才算真正的开始。
而近代设计一味重视设计带来的知性美，忽略了人们对生物的感情，这样的设计理念从根本上来说还是更重视产品的生产性和使用功能，主张禁欲之美。但却忽略了产品应该带给人的触感。
素材感觉表达了回归素材的思想，而回归素材也意味着回归人类最原始的感觉。在日本人的审美意识中，素材的重要性要远远大于外观。

内涵丰富的极简主义

欧洲文化给人的印象总是在竭尽全力地试图征服自然，与此不同，日本文化则主张与自然融为一体。在日本人的意识中，人类和自然是一体的，同时尊重自然本来的模样。“less is more”，是一种提倡简单，反对过度装饰的设计理念，在注重丰富的素材感与存在感之后，形态就变得非常简单了。总而言之，由于日本人对于丰富素材的过分重视导致了对形态意识的越来越淡，因而孕育了日本独特的极简形态文化。
内涵丰富的极简主义即先有丰富的素材，然后产生了零星的形状，为了充分利用素材所蕴含的丰富性，于是保持了物品原有的形态。也就是通过简化形态来强调素材本身的设计表现形式。

负创作

说起创造，人们经常将其看成是一个从无到有的过程，但事实上，创造是一个去除人们内心对某一事物既有概念、过度思考等存在于人脑中的各种想法的过程。如果人们的思考方式一味的被过去的思考方式和过去事物的面貌所束缚的话，那么就无法真正的创作。也就是说，创造也是一种从过去解放自己的行为。
所谓的负创作，就是在种种的矛盾中探索事物真正的本质，去除不必要的解释说明，回归事物本身真实的状态。通过删除的方式，用最简洁的表现形式，解释物体终极的存在意义。
如果用减法的方式做设计，得到的效果可能略显沉默但是，较之做加法的设计，它却更能激发人的想象力，因为删减之后的设计引起的是人们对其的违和感和好奇心，而不是本能的共鸣，用刺激引发人们的思索，这种表现不是单纯的传达，而是刺激。

阴影的美

夕阳的美在于逆光，人们站在夕阳下，身后出现阴影，而阴影又被阳光反射，产生了逆光，就像海岛的影子和云彩，穿上一层光亮的轮廓，产生了独有的朦胧美。
了解和感动是两种不同的情感，大多数人认为照明不过就是使物品和空间“变得明亮”，然而事情却不尽如此。正如逆光的作用就是创造阴影。虽然顺光将物体清晰明白的呈现在人们的眼前，也很重要，却无法唤起人们内心的感动。
在日本，无论是茶室还是民宅，日本的建筑文化都体现在对采光隔扇上映出阴影的理解。民宅中的各种阴影，住宅深处微暗的的空间，光照在采光隔扇上形成的逆光，造就了独有的朦胧美，也只有在民宅的阴影中，金屏风和漆器的碗才彰显华美。
还有很多种其它的光，虽然不是逆光的程度，也能够给人以感动，比如有一种光在明亮的墙壁上将人和自然演绎出剪影的风格，让人们看到的是整个空间而不是某一具体的事物。

多线程之GCD(上篇)

post @ 2016-05-14

多线程之GCD(上篇)

什么是GCD

GCD（Grand Central Dispatch）是异步任务的技术之一。
在没有GCD之前，我们可以使用Cocoa框架中NSObject类的方法去处理线程问题，用performSelectorInBackground:withObject实例方法用来执行后台线程，用performSelectInSelectorOnMainThread实例方法来使用主线程。虽然performSelector系方法比使用NSTread类进行多线程要简单，但与GCD相比，它不够简洁。

- (void)viewDidLoad {
[super viewDidLoad];
// 后台线程执行
[self performSelector:@selector(clickSelector:) withObject:nil];
// 主线程执行
[self performSelectorOnMainThread:@selector(clickSelector:) withObject:nil waitUntilDone:NO];
}

为什么要使用GCD：

应用程序启动时，通过主线程来描绘用户界面、处理触摸屏幕的事件，如果在该线程中处理耗时的操作，比如图像处理和数据库访问，就会妨碍主线程的执行(这个主线程也就是Runloop)，从而导致不能更新界面、程序画面长时间停滞等问题，为了解决这个问题，就需要将这些耗时的操作放到其它线程中执行。
我们现在一个物理的CPU芯片实际上有64个（64核）CPU，而一个CPU核执行的命令为一条无分叉路径，也就是一个线程。GCD就是是苹果为多核的并行运算提出的解决方案，它会自动合理地利用更多的CPU内核（比如双核、四核），最重要的是它会自动管理线程的生命周期（创建线程、调度任务、销毁线程），完全不需要我们管理。

GCD的使用

dispatch_async(myConcurrentDispatchQueue, ^{
    NSLog(@"my block");//耗时的操作
});

在block中定义要执行的任务，通过dispatch_async追加在变量queue中，这里的queue（Dispath Queue）就是执行处理的等待队列，它分为下面两种：
串行: Serial Dispatch Queue,等待现在的执行中处理结束。
并行： Concurrent Dispatch Queue ，不等待现在执行中处理结束。

创建Serial Dispatch Queue的方法：

dispatch_queue_t mySerialDispatchQueue =     dispatch_queue_create("com.gcd.mySerialDispatchQueue", NULL);

Serial Dispatch Queue中每次只能追加一个处理，它可以用来解决数据竞争的问题。但如果生成多个Serial Dispatch Queue，它们将并行执行，当然也会消耗大量内存，引起大量的上下文切换，应该生成仅限所必须的个数。当想并行执行不发生数据竞争等问题的处理时，使用Concurrent Dispatch Queue，无论生成多少个，XUN内核会有效管理线程。

创建Concurrent Dispatch Queue的方法：

dispatch_queue_t myConcurrentDispatchQueue = dispatch_queue_create("com.gcd.myConcurrentDiapatchQueue", DISPATCH_QUEUE_CONCURRENT);
 dispatch_async(myConcurrentDispatchQueue, ^{
     NSLog(@"my block");//耗时的操作
 });

对于dispatch_asyn的内存管理，需要注意，在dispatch_asyn函数中追加Block到Dispatch Queue后，即使立即释放Dispatch Queue，由于它被Block所持有也不会被马上废弃，因而block能够执行，block执行结束后会释放Dispatch Queue,当不被持有也就被废弃了。

Main Dispatch Queue 和 Globle Dispatch Queue

即便我们不自己生成dispatch Queue，系统也会提供给我们几个，那就是Main Dispatch Queue和Globle Dispatch Queue。顾名思义，Main Dispatch Queue也是在主线程中执行的Dispatch Queue，因为主线程
只有一个，它必然也就是Serial Dispatch Queue，追加到Main Dispatch Queue的处理在主线程的Runloop中执行。另一个Globle Dispatch Queue是所有应用程序都能够使用的Concurrent Dispatch Queue，它有四个执行优先级，分别是：Hight Priority、Default Priority、Low Priority和Background Priority,在向Globle Dispatch Queue追加处理时，应选择与处理内容对应的执行优先级的Globle Dispatch Queue。

各种Dispatch Queue的获取方法：

// Main Dispatch Queue 的获取方法
dispatch_queue_t mainDispatchQueue = dispatch_get_main_queue();

// Globle Dispatch Queue （高优先级）的获取方法
dispatch_queue_t globleDispatchQueueHigh = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_HIGH, 0);

// Globle Dispatch Queue （默认优先级）的获取方法
dispatch_queue_t globleDispatchQueueDefault = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT, 0);

// Globle Dispatch Queue （低优先级）的获取方法
dispatch_queue_t globleDispatchQueueLow = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_LOW, 0);

// Globle Dispatch Queue （默认优先级）的获取方法
dispatch_queue_t globleDispatchQueueBackground =     dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_BACKGROUND, 0);

变更执行优先级：dispatch_set_target_queue

// 如果将多个串行的queue指定到了同一目标，那么着多个串行queue在目标queue上就是同步执行的，不再是并行执行。
dispatch_queue_t targetQueue = dispatch_queue_create("test.target.queue", DISPATCH_QUEUE_SERIAL);  
dispatch_queue_t queue1 = dispatch_queue_create("test.1", DISPATCH_QUEUE_SERIAL);
dispatch_queue_t queue2 = dispatch_queue_create("test.2", DISPATCH_QUEUE_SERIAL);
dispatch_queue_t queue3 = dispatch_queue_create("test.3", DISPATCH_QUEUE_SERIAL);

dispatch_set_target_queue(queue1, targetQueue);
dispatch_set_target_queue(queue2, targetQueue);
dispatch_set_target_queue(queue3, targetQueue);

延迟后执行：dispatch_after

// 3秒钟后执行操作
dispatch_time_t time = dispatch_time(DISPATCH_TIME_NOW, 3ull * NSEC_PER_SEC);
dispatch_after(time, dispatch_get_main_queue(), ^{

});

第一个参数为指定的开始时间，第二个参数是指定的时间单位后。NSEC_PER_SEC是以秒为单位，NSEC_PER_MSEC是以微秒为单位。
dispatch_after并不是在指定时间后执行处理，而只在指定时间追加处理到Dispatch Queue,因为Main Dispatch Queue在Runloop中执行，所以当它有大量的处理追加的时候，会有延迟的时间，如果不是对时间特别的精准，大致的延迟效果是很不错的。

Dispatch Group

经常有一种情况，我们需要在追加到Dispatch Queue中的多个处理全部结束后进行处理，可以使用Dispatch Group。

dispatch_queue_t queue = dispatch_get_global_queue(DISPATCH_QUEUE_PRIORITY_DEFAULT, 0);
dispatch_group_t group = dispatch_group_create();
dispatch_group_async(group, queue, ^{
    NSLog(@"block1");
});
dispatch_group_async(group, queue, ^{
    NSLog(@"block2");
});
dispatch_group_async(group, queue, ^{
    NSLog(@"block3");
});
dispatch_group_notify(group, dispatch_get_main_queue(), ^{
    NSLog(@"执行完成");

});

最后的执行结果：

block1
block2
block3
执行完成

因为多个线程并行执行，所以追加的顺序不定。

Dispatch_barrier_async

dispatch_queue_t queueBarrer = dispatch_queue_create("com.concurrentDiaptchQueue.forBarrier", DISPATCH_QUEUE_CONCURRENT);
dispatch_async(queueBarrer, blcok1);
dispatch_async(queueBarrer, blcok2);
dispatch_async(queueBarrer, blcok3);
//dispatch_barrier_sync会将上面追加的并行操作全部处理结束之后，再将dispatch_barrier_sync的处理追加上，然后由dispatch_barrier_sync函数追加的处理全部执行完毕后，才恢复下面的动作。
dispatch_barrier_sync(queueBarrer, blaock6);
dispatch_async(queueBarrer, blcok4);
dispatch_async(queueBarrer, blcok5);

当访问数据库文件时，用Serial Dispatch Queue可以避免数据竞争的问题，因为读取操作与写入操作不能惊醒并行处理。但是为了高效，我们可以只并行的进行写入处理，并行的进行读取处理，那么多个并行执行就不会发生问题。
